Your benchmark says 0.2 ns/op

Your benchmark says 0.2 ns/op.
That number looks precise.
Most of the time, it doesn’t mean what you think it means at all.

The illusion of precision

You wrote a benchmark.

You ran it.

You got a number.

And it looks… scientific.

But precision is not correctness.

A number can be precise and still completely meaningless.

You are not benchmarking your code

At nanosecond scale, you are not measuring a function.
You are measuring a system.

  • the JVM
  • the JIT compiler
  • the CPU
  • the operating system
Each layer interferes with your measurement.
Each layer can change what is actually executed.

The code you think you run may not exist

Consider this:

                long start = System.nanoTime();
                doSomething();
                long end = System.nanoTime();
Looks reasonable.

It isn’t.

The JVM is allowed to rewrite your benchmark

The JIT compiler is not here to respect your code. It is here to optimize it.
  • methods get inlined
  • computations get simplified
  • unused results get removed
In extreme cases:

your code is not optimized.
it is erased.

Dead code elimination

If the result of your computation is not used:

the JVM may remove it entirely.
You are benchmarking a void.

Constant folding

If a result can be computed ahead of time:

the JVM may replace your computation with a constant.

Warmup effects

At first, your code runs interpreted.

Then it gets compiled.

Then optimized.

Measuring too early means measuring the wrong thing.

The three benchmark killers

  • Dead Code Elimination → your code never runs
  • Constant Folding → the result is precomputed
  • Warmup → you measure interpretation, not optimized execution

Even when the code runs, you measure noise

Let’s assume your code actually executes.

You are still not measuring just your logic.

You are measuring:
  • CPU cache behavior
  • branch prediction
  • frequency scaling
  • OS scheduling
At nanosecond scale, these dominate.

A reality check

A main memory access is ~100 ns.

If your benchmark reports 0.2 ns:

you are not measuring business logic.
You are measuring:
  • L1 cache behavior
  • CPU pipelines
  • or noise

The deeper problem: averages lie

Even if your measurement was correct…

A single number is not enough.

Most naive benchmarks report:

average latency
This is almost always misleading.

Why averages are dangerous

Imagine this distribution:
  • 99 requests: 1 ms
  • 1 request: 500 ms

Average ≈ 6 ms

Looks fine.

Reality:

one request is catastrophically slow.

This is why percentiles matter

Instead of averages, you need distributions:
  • P50 → typical case
  • P95 → high latency
  • P99 → worst-case under normal load
Systems fail at the tail.
Not at the average.

Microbenchmarks vs real systems

This is where confusion happens.

Microbenchmarks (JMH)

  • measure isolated code
  • focus on throughput / cost per operation
  • require strict methodology

Production systems

  • measure latency distributions
  • care about tail behavior (P99, P999)
  • involve IO, contention, queuing
A microbenchmark tells you how fast a function can run.
A percentile tells you how your system actually behaves.

Why JMH exists

Serious benchmarking requires discipline.

JMH exists to enforce it:
  • warmup phases
  • multiple forks
  • isolation from JVM state
  • statistical analysis
But even JMH is not magic.
It reduces error.
It does not remove the need for understanding.

The real skill

Benchmarking is not about writing code.

It is about asking:
What am I actually measuring?
And:
Is this number representative of reality?

Final insight


At small scales, measurement is fragile.

At system scale, measurement is incomplete.

The number is precise.
The system is not.

Conclusion

If you don’t understand the runtime:
  • your benchmarks are misleading
  • your conclusions are fragile
  • your optimizations are often wrong
You think you are measuring code.
You are measuring behavior.
