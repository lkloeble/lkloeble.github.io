The JVM Can Legally Produce Impossible States

Two threads.
Two variables.
No locks.

At first glance, the following outcome looks impossible:

r1 = 0
r2 = 0

And yet, under the Java Memory Model (JMM), this result is perfectly legal.

This article explains why.

If you want to see it happen yourself, a small demo program is available here:

https://github.com/lkloeble/jmm-impossible-states-demo

The scenario

Consider the following code executed by two threads.


                    Thread 1:

                    x = 1;
                    r1 = y;

                    Thread 2:

                    y = 1;
                    r2 = x;
Both shared variables start at `0`.

Intuition suggests that at least one read should observe the other thread’s write.

Yet the following outcome can occur:

r1 = 0
r2 = 0

At first sight, this appears to violate causality.

It does not.

The key rule: no happens-before, no guarantees

The Java Memory Model defines when writes performed by one thread must become visible to another thread.

That guarantee is established through a happens-before relationship.

Typical constructs that create happens-before include:

  • `volatile`
  • `synchronized`
  • `Lock`
  • thread start/join
  • concurrent data structures
If no such relationship exists, the JVM provides very few guarantees about:
  • ordering of operations
  • visibility of writes
  • timing of memory propagation
And our example deliberately contains no synchronization at all.

Instruction reordering

One reason the paradox occurs is instruction reordering.

The JVM and the CPU are allowed to rearrange instructions in order to optimize execution, as long as the result appears correct for a single thread.

For example, the following transformation is legal: 


                    Thread 1:

                    r1 = y
                    x = 1

                    Thread 2:

                    r2 = x
                    y = 1

If both reads happen before the writes become visible, both threads observe `0`.

From each thread’s point of view, the execution still appears valid.

The paradox only emerges when we combine both observations.

Sequential consistency illusion

A subtle point often missed is that each thread still observes its own instructions in program order.

The JVM preserves what looks like sequential execution for each individual thread.

The illusion breaks only when multiple threads observe the same memory.

This is why concurrency bugs often appear so mysterious:
the code seems perfectly correct when reasoned about locally.

The hardware reality: store buffers


The explanation goes deeper than compiler optimizations.

Modern CPUs do not immediately write data to main memory.

Instead, writes are placed in store buffers.

A simplified view looks like this:

Thread → Store Buffer → Cache → Main Memory

When a thread executes:

x = 1

the value may first be placed in the CPU’s store buffer.

From the perspective of the writing thread, the write has already happened.

But other cores may not see it yet.

This means another thread may still read the old value.

Store buffers exist because they allow CPUs to:

  • reduce memory latency
  • pipeline operations
  • maintain high throughput
Without them, modern processors would be dramatically slower.

But they also make memory visibility more complex.

Why the demo works

The demo program repeatedly executes the scenario until it observes:

r1 = 0
r2 = 0

When this happens, it prints something like:

Observed the "impossible" state:
r1 = 0, r2 = 0
First observed after 26053 iterations
Elapsed time: 1315 ms

This means the program detected the outcome for the first time after 26,053 attempts.

Not that it happened 26,053 times.

The result depends heavily on:

  • CPU architecture
  • JVM implementation
  • thread scheduling
  • system load
  • pure timing luck

Sometimes it appears quickly.
Sometimes it takes much longer.

Restoring order: establishing happens-before


The key lesson is that the problem disappears as soon as we introduce proper synchronization.

Multithreading is not chaos.
It only becomes chaos when we remove the rules that make memory visible and ordered.

For example, if the shared variables are declared `volatile`:


                    volatile int x = 0;
                    volatile int y = 0;

                    Thread 1:
                    x = 1;
                    r1 = y;

                    Thread 2:
                    y = 1;
                    r2 = x;

`volatile` introduces memory barriers that enforce two important guarantees:

1. writes to a volatile variable are immediately visible to other threads
2. certain instruction reorderings are forbidden

Once those guarantees exist, the `0 / 0` outcome is no longer possible.

In other words, the chaos disappears once we establish a formal happens-before relationship.

Java provides several tools to establish that ordering:
  • `volatile`
  • `synchronized`
  • locks
  • concurrent collections
  • atomic variables

Without them, multithreaded code operates in a world where ordering and visibility are intentionally relaxed for performance.

Why this matters in real systems

In small demo programs, this behavior is fascinating.

In large production systems, it can be dangerous.

In legacy monoliths especially, concurrency issues often remain dormant for years.

They may appear only when:

  • the system reaches higher load
  • the hardware changes
  • the JVM version changes
  • the deployment environment changes

For example, moving from x86 to ARM can expose hidden concurrency bugs.

x86 provides relatively strong memory ordering guarantees.

ARM allows much more aggressive reordering.

Code that "worked for years" may suddenly start behaving unpredictably.

The real lesson

Multithreading is not chaos.

But without synchronization, it can look like it.

The Java Memory Model deliberately relaxes ordering and visibility rules in order to allow modern hardware and JIT compilers to run efficiently.

If two threads communicate through shared state, that communication must be explicit.

Otherwise the JVM is free to produce results that seem impossible — but are perfectly legal.

Final thought

Concurrency bugs rarely appear when you are looking for them.

They appear under load, on a different machine, or months after a harmless-looking change.

In large Java systems, diagnosing those behaviors requires understanding not only the code, but also the memory model and the hardware beneath it.
